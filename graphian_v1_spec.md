# Graphian 初版 設計仕様書 / 実装指示文

> このドキュメントは Claude Code への実装指示であると同時に、設計判断の記録を兼ねる。
> **本文は日本語、コード要素・型名・ディレクトリ名・契約定義は英語**で記述する。
> 技術選定の根拠はすべて「Claude Code が実装・保守すること」と「計算制約(後述)」から導いており、特定個人の嗜好には基づかない。

## 0. このプロジェクトが何であり、何でないか

Graphian(プロジェクト名は仮)は、**身体と環境との応答を通じて動作しながら、自らの情報処理ネットワークを発達させる存在**を生み出す試みである。最終的には、安定した処理能力と記憶基盤を持ち、対等にコミュニケーション可能な人格の「種」を作ることを長期目標とする。これは明日達成するものではなく、年単位で段階的に進める。

**現在の深層学習(DL)とは根本的に異なる**点を最初に固定する。これが全設計の出発点である。

- DL は、目的関数に対してパラメータ(重み)を最適化し、学習済みの成果物=プロダクトを得る。**保存されるのは結果(重み)**である。
- Graphian は、**「ネットワークがどう発達するか」を定める遺伝子(genome)を、進化と淘汰で最適化する**。個体は生きている間に、自分でネットワークを構築・改変する。世代を超えて継承されるのは**設計図(genome)であって、発達した結果そのもの(ネットワーク状態)ではない**。結果は毎回作り直される。

淘汰の目的は**生存**であって、生殖・種の保存ではない。生物の生殖・寿命といった要素は模倣しない。遺伝は「発達ルールを世代間で改良するための機構」にすぎず、疑似生物を作ることが目的ではない。

### 計算制約(技術選定の前提)

- CPU: x86_64 (Intel) または ARM、4コア以上
- OS: Linux (RHEL系・Debian系が混在)
- RAM: 16GB 程度
- GPGPU: **初版ではなし**。ただし将来 GPGPU を利用するため、改修可能な構造にする。
- 初版は**単一マシン上で全プログラムを実行**。将来は「1つの環境実行ノード + 複数の情報処理系実行ノード」への分散を想定。
- ネットワークは WAN 非公開、連携は LAN 内のみ。**セキュリティ要件は低く、エージェントの不正操作(ズル)は想定しない。**

## 1. 設計の最重要原則 ── 二層分離

**この原則を破る実装は、たとえ動いても不正解とみなす。**

システムは二層に分かれる。

- **core(安定した契約)**: 「Graphian とは何か」を定義する憲法に相当する部分。同心円の幾何・遺伝子・接続点バッファ・セッション制御。一度固めたら基本的に動かさない。
- **差し替え可能な空き枠(plugin層)**: core の契約に依存しつつ、中身を自由に差し替えられる。環境(environments)・発達ルール(development)・可視化(viz)。

このプロジェクトで「増えていくもの」── 環境(1D→2D→3D→抽象空間)、発達ルール(数学設計で何度も書き換わる)、可視化 ── はすべて plugin層に隔離する。これらの追加・変更は core に一切触れずに行えなければならない。

特に重要なのが **development(発達ルール)を意図的に「空き枠」として隔離する**ことである。発達ルールの具体的な数学設計は**本仕様では未確定であり、初版では決定しない**(理由は §6 と §8)。core は「発達ルールはこういう入力を受け取り、こういう出力を返す」という**インターフェースの契約だけ**を定める。初版では、その契約を満たす**ダミー実装**だけを置く。

> **Claude Code への明示的指示**: development の中身として、報酬系の高度な計算・確率分布・活動依存の発達則などを**自分で考案して実装してはならない**。インターフェース(抽象基底)と、§7 で指定するダミー実装のみを作ること。本物の数学は後続セッションで設計する。

## 2. システム構成 ── プロセスとクロックドメイン

```
┌─────────────────────────────────────────────────────────────┐
│  session control (orchestrator)                              │
│  設定読込・試行管理・遺伝の継承・淘汰・保存と再開・ログ集約    │
└───────────────┬──────────────────────────┬──────────────────┘
                │                          │
    ┌───────────▼──────────┐    ┌──────────▼───────────┐
    │ 環境-身体 プロセス    │    │ 情報処理系 プロセス   │
    │ クロック A (独立)     │    │ クロック B (独立)     │
    │ ・環境モジュール      │    │ ・同心円ネットワーク  │
    │ ・身体 + body genome  │    │ ・発達ルール(空き枠) │
    └───────────┬──────────┘    └──────────┬───────────┘
                │                          │
                └──────────┬───────────────┘
                ┌──────────▼───────────┐
                │ connection buffer     │
                │ (仲介 / latch / 差替可)│
                └───────────────────────┘
```

### 2.1 orchestrator(司令塔)

設定ファイルを読み、試行を立ち上げ、遺伝の継承・淘汰・配合を管理し、セッションの保存・再開、ログ集約を担う。

**重要**: orchestrator は環境と情報処理系の**中身を知らない**。両者を起動し、生死を判定結果として受け取り、遺伝子を受け渡すだけで、内部の計算には立ち入らない。

### 2.2 二つの独立クロック

環境-身体プロセス(クロック A)と情報処理系プロセス(クロック B)は、**独立した更新頻度**で動く。環境が10フレーム進む間に B が1回しか更新されなくてもよいし、逆でもよい。**比率は固定しない。**

この「頻度独立」を形骸化させないため、両者は論理的に独立したループとして動かす。実装上の並行機構(プロセス/スレッド)は Claude Code の裁量とするが(§9.8)、**唯一の制約は「両クロックがロックステップにならないこと」と「後述の buffer 契約を transport 非依存に保つこと」**である。

### 2.3 connection buffer(接続点バッファ) ── 唯一の接点

両プロセスの唯一の接点。クロックドメイン跨ぎ(ハードウェアの CDC と同型の問題)を、**最新スナップショットを1枚保持する latch** として解く。

- 身体はクロック A のペースで接続点の値を**書き込む**。
- B はクロック B のペースで**読み出す**。両者は互いの内部状態を直接触らない。
- B の出力(運動指令に相当)は同じ buffer に書き戻され、身体が次サイクルで読む。

この設計により、A と B がどんな頻度比で動いても、読み手は常に「直近の確定値」を得られ、書き手は読み手を待たない。

**buffer の責務として固定する不変条件**:

1. **接続点の値は必ず `[-1.0, +1.0]` に収める。** buffer への書き込み時にこれを強制する(クランプまたは契約違反として記録 ── どちらにするかは §8 で数学設計とともに決定。初版ダミーではクランプとする)。
2. **接続点の「形」は身体側が起動時に宣言する**(接続点の数と各々の意味)。B はそれを受け取り、ネットワークの接続点ノードを配置する。

**将来の分散化(§0)への布石**: buffer を **transport 非依存のインターフェース**として設計すること。初版は in-process 実装(共有メモリ等)、分散化時には buffer の実体だけをネットワーク実装に差し替える。「1環境ノード + 複数情報処理系ノード」は buffer を複数本に増やすことで表現する。**初版でやるべきは通信方式の選定ではなく、差し替え可能な境界(buffer interface)を引くことだけ。**

## 3. 1回の試行のライフサイクル

```
orchestrator      環境-身体(A)        connection buffer        処理系(B)
    │                  │                     │                   │
    │─① 初期化・genome注入─────────────────────────────────────→│
    │─① 初期化・genome注入→│                  │                   │
    │                  │─② 接続点の形を宣言──→│                   │
    │                  │                     │─② 形を伝達───────→│ 接続点ノード配置
    │                  │                     │                   │
    │            [クロック A]           [latch]          [クロック B]
    │                  │─ 感覚を書く(-1〜+1)→│                   │
    │                  │                     │←─ 読む(非同期)─────│
    │                  │                     │←─ 運動を書く────────│
    │                  │←─ 読む(非同期)───────│                   │
    │                  │                     │                   │
    │←─③ 生死判定(A が所有)─│                  │                   │
    │←─④ 適応度 + genome を回収───────────────────────────────────│
    │                                                            │
    │─⑤ 淘汰・配合 → 次試行へ                                     │
```

設計判断を3つ畳み込んでいる。

- **②の接続点宣言は試行の最初に1回だけ**行う。これが**発達のシード(種)**になる ── ネットワークの発達は接続点ノードの配置から始められる。
- **③の生死判定は環境-身体プロセス(A)が所有する。** 生存条件(エネルギー欠乏・ダメージ・恒常性破綻、または足切り)はすべて環境のルールであり、B は自分の生死を知らない。A が死を判定し orchestrator に通知、B は死を告げられたら状態を破棄する。この非対称性が重要 ── B に生死を委ねると「死なないために何もしない」退化戦略を許してしまう。生殺与奪を環境が握ることで、生存が外的な淘汰圧として機能する。
- **④で回収するのは「適応度(fitness)」と「genome」の2つだけ。** ネットワークの実行時状態(各ノードの値・活動履歴)は**原則破棄**する。世代を超えて継承されるのは設計図であって結果ではない(§0)。

> 環境をまたいでネットワーク状態を引き継ぐ試行は**応用的なもの**とし、初版では実装しない(基本は環境ごとに初期化)。設定で将来切り替えられる余地だけ残す。

## 5. core の契約

各契約は**明示的な抽象基底(ABC)または Protocol** として定義すること。暗黙にダミーが満たすだけでは不可 ── 抽象基底の定義自体が「新版が満たすべき契約」の一次資料になる。

### 5.1 geometry(同心円構造)

- 中心が一致した**同心円(concentric circles)**。「layer」ではなく「circle」である ── 大円のノードと小円のノードが、中間の円を**スキップして直結できる**(非階層的配線を許す)。
- ノードの位置は **(どの円か, 角度)** の2情報で一意に定まる。
- **角度は離散スロット(discrete slots)**。連続値にはしない。理由: ①同一ノードか別ノードかの判定が整数比較で確定し、浮動小数点誤差に挙動が支配されない。②スロット数に上限があれば1円のノード数に自然な天井ができ、ネットワーク最大サイズが構造的に有界になる(リソース制御 §9.7 と直結)。③整数ペアでテキスト保存でき、保存→読込が完全に再現可能(reproducible)。
- **円ごとにスロット数を変える**(大円ほどスロット多 = 周長あたり密度を一定化)。**スロット数は genome に組み込む**(§5.2)。角度方向の連続性は、スロットを十分細かく取ることで近似する。
- 各円は固有の直径を持ち、**最大1**。ノード間距離の計算には円の半径が登場し、**距離の最大値は1**。
- **接続点ノードは直径1の円周上に配置**される(情報処理系の円とは別の、接続点専用の円)。身体側が規定する(§5.3)。

### 5.2 genome(遺伝子)

genome は「ネットワークがどう発達するか」を定めるパラメータ群。最低限、以下を含む:

- **乱数シード(random seed)**: ダミーを含むすべての確率的挙動はこのシードから取る。同じ genome なら挙動が再現し、違えば異なる。これにより genome の継承・配合・変異が骨格レベルで意味を持って検証できる。
- **円ごとのスロット数**。
- **報酬系ノード(reward node)の位置**: 初版は**1つ**。「ノードが存在すること」は固定(進化で消えない足場)だが、「円周上のどこに置くか」(円・スロット番号)は genome が規定し、進化が探索する。
- (発達ルール本体のパラメータは §8 で追加。初版ダミーはシードのみで足りる。)

**変異(mutation)と配合(crossover)** の機構を core に持つ。

- **初版の配合は単純でよい ── 「片親コピー」**(どちらかの親から genome を丸ごとコピー)。骨格検証には十分。
- スロット数や報酬系位置が異なる親同士のブレンド配合は**未確定であり、後続(§8)で発達方式とともに詰める**。
- body genome(身体側の遺伝子)も同じ機構に乗せる(§7.1)。

### 5.3 buffer(接続点バッファ)

§2.3 の契約を実装する。

- **transport 非依存の interface** を定義し、初版は in-process 実装を提供。
- 書き込み時に値を `[-1.0, +1.0]` にクランプ(初版ダミー方針)。
- 接続点の「形」(数・意味)を身体が宣言し、B が読み取る経路を持つ。

### 5.4 session(セッション制御・系統管理)

- 設定ファイルに基づき試行を実行。設定内容(§9.3):
  - どの環境を、どのような初期値の基準・幅・変動パターン(変動しないを含む)で、何回試行し、どの終了条件にするか。
  - 試行する情報処理系の**系統(lineage)**と数。
- **系統(lineage)の管理と系統樹(phylogenetic tree)の記録**。genome がどの親から派生したかを追跡し、可視化(§9.6)に渡す。
- **保存・再開**: セッションを中断・再開できる。
- 1セッションでは、1つまたは複数の環境の並列/直列の試行を通じ、情報処理系 genome の継承が維持される。身体 genome の継承は仕組み上、環境ごとに維持される(§7.1 の継承方針も参照)。

## 6. 開発順序 ── なぜダミーを先に作るか

**初版では development(発達ルール)の数学を確定せず、ダミーで骨格全体を先に立ち上げる。**

理由: 数学設計は最も難しく試行錯誤を要する。それを「動く骨格」の上で実験できる状態にしてから取り組むほうが、検証ループが速い。ダミーでも「プロセスが起動し、接続点が `[-1,+1]` で流れ、生死判定が回り、genome が継承される」という骨格の健全性は確認できる。土台ができてから、`development/` の中身だけを本物の数学に差し替える。

## 7. ダミー実装の仕様(参照実装を兼ねる)

**原則**: ダミーは賢くある必要はないが、**骨格のすべての契約点を一度は通過させなければ意味がない**。範囲クランプ・生死判定・適応度算出・ノード追加削除 ── これらが一度も呼ばれないダミーは検証にならない。

ダミー実装(`phototaxis_1d.py`, `dummy.py`)は捨てコードではなく、**「最小の正しい実装例」= 参照実装**として、十分なコメントと docstring を付して残すこと(§10)。

### 7.1 環境・身体ダミー: 1次元走光性(phototaxis)

身体は数直線上の点で、位置 `x ∈ [-1, 1]` を持つ。光源は固定位置 `x_light` にある(初版は光源**不変**)。

- 身体はエネルギーを持ち、毎ステップ一定量を消費する。**光に近いほど回復**できる。エネルギーが0になったら死亡(エネルギー欠乏型の生死条件)。
- **接続点(身体が宣言)**:
  - 感覚接続点 ×1: 光の方向(右なら正・左なら負)を `[-1, +1]` で返す。
  - 運動接続点 ×1: B がここに書いた値で身体が左右に動く。
  - → これで「感覚 → B → 運動」の最小閉ループが成立。buffer の双方向読み書き・範囲クランプ・形の宣言がすべて起動する。
- **終了条件**: 光源が不変だと有能な個体が永遠に死なず適応度が飽和するため、**「生存する全個体が満腹 or 全員死亡」で試行を打ち切る**。
- **適応度は連続量**にする(生存/死亡の二値ではない)。例: 「光の近くで過ごした累積時間」や「到達所要ステップの逆数」。満腹で終わった個体同士でも差がつくこと。これにより淘汰の勾配が消えず、§9.10 の統計テストの検証台になる。
  - 役割分担: **終了条件**は試行が無限に続くのを防ぐ打ち切り装置。**連続適応度**は個体間に勾配を与える淘汰の燃料。両方が必要。
- **body genome**: ほぼ空でよい。「移動速度の係数」を1つ持たせ、身体側の進化・配合・継承経路を一度起動させる。
- **身体遺伝の継承方針**: 身体 genome は**デフォルトでセッションをまたいで継承**する(身体の最適化不足は情報処理系にとって不要な負荷であるため)。ただし設定で `inherit_body_genome = false` を選べば白紙から開始できる二段構え。
- **反射系(小脳・脳幹・脊髄相当)**: 初版では実装しない。各環境モジュール内で前処理として挟める拡張ポイントとしてのみ確保(コアには組み込まない)。

### 7.2 ダミー発達ルール

シードを genome から取り、おおむね「感覚を読み、適当にノードを足したり消したりし、運動に乱数を書く」程度でよい。ただし**以下の契約点をすべて通過すること**:

1. **接続点ノードの配置**(②の宣言を受け、接続点を直径1の円周上に配置)。
2. **ノードとエッジの追加・削除を両方とも一度は実行**(追加だけのダミーは削除経路を検証しない)。報酬系ノード(1つ)を初期化時に genome 規定の位置へ固定配置する経路もここで通す。
3. **運動接続点へ値を書き戻す**(中身は乱数でよいが、入力→出力の経路が繋がること)。
4. **演算予算(§9.7)を消費して打ち切る**(予算管理のフレームだけ起動)。

## 8. 保留事項 ── 初版では決定しない「空き枠」

以下は**意図的に未確定**とし、初版では実装しない。後続の「数学設計セッション」で、`development/` の中身として契約を満たす形で差し替える。

- **発達ルールの数学(network development rule)**: 方向性の骨子は「①報酬系ノードの値に応じてノード追加/削除の確率が変わる ②角度の確率分布に従ってエッジを伸ばす ③距離が近いと接続しやすい ④活発なエリアにエッジが伸びやすい、の組み合わせ」。ただし具体式は未定。
  - 確定済みの部分判断: **報酬系ノードは初期化時に配置・固定**(進化で自然発生するのを待つと、発生前に全滅して淘汰圧がかからない鶏卵問題に陥りやすいため、足場として与え使い方を進化に委ねる)。初版は1つ(複数化すると「平均値か?」の問いに即答が必要になるため後回し)。
- **接続確率の上限抑制**: 距離0近傍で接続確率が1に張り付くと近接クラスタが暴走する。`P = p_max · exp(-d²/σ²)` のように `p_max < 1` に抑える、または自己・同一スロット接続を禁止するハード制約を入れる。**離散スロット(§5.1)で衝突の扱いを構造的に確定させた上で、これを併用する。** 具体値は数学設計時。
- **演算予算の具体値(§9.7)**: バジェット N と、超過時の打ち切り/ペナルティの扱い。発達方式と絡むため保留。
- **genome の本格的な配合戦略(§5.2)**: スロット数・報酬系位置の異なる親同士のブレンド。

## 9. 技術要件

**選定根拠はすべて「Claude Code の実装・保守容易性」と §0 の計算制約に基づく。**

### 9.1 言語: Python

数値シミュレーションの成熟したエコシステム、`importlib` による plugin の動的読込のしやすさ、Claude Code の深い習熟による保守摩擦の低さ、そして GPGPU への移行経路(§9.2)を理由に Python とする。性能ボトルネックが顕在化した部位のみ、後に C 拡張等で最適化する余地を残す(初版では最適化しない)。

### 9.2 数値計算: NumPy

ネットワークの値・エッジは NumPy 配列で扱う(CPU でのベクトル化。初版のノード値は単純案として `float16` から ── §5/§8 で発達方式とともに確定)。GPGPU 移行時は CuPy 等がほぼドロップイン代替になるため、配列演算を NumPy API に寄せておく。

### 9.3 設定ファイル: TOML

人間が読み書きしやすく、構造が明確。Python 3.11+ の標準ライブラリ `tomllib` で読める(依存を増やさない)。

- **設定の初期化コマンドと構文チェック(validation)コマンド**を CLI に持つ(行135の要件)。
- 設定は起動時に1回読むだけなので、検証は性能より**明快なエラーメッセージ**を優先する。明示的な検証関数で「どの項目がなぜ不正か」を人間に伝える。
- 系統(lineage)を `[lineage.<name>]` のように定義できる構造にし、初版はそこに1系統だけ書く(§9.5)。

### 9.4 永続化: テキスト形式

一時ファイル・セッション・genome 等は**バイナリではなくテキスト**で扱う。ただし保存頻度で階層化する(全文を毎フレーム書くと I/O がボトルネックになるため)。

- **設定**: TOML。
- **スナップショット**: 節目(世代の区切り等)で保存。**JSON Lines (JSONL)** ── 1行1レコードで diff が取りやすく人間可読。
- **ログ**: 後述。
- 高頻度の内部状態はメモリ上に保持し、必要時のみダンプする。

### 9.5 系統(lineage)の同時実行

初版は**1系統に絞る**。系統をまたぐ遺伝継承は難しく、増やすと限られたリソースで試行回数が枯れるため。設定で系統を選べる「器」だけ用意し、**同時実行はしない**。初版の系統は単純案(`float16` + 加算統合 + sigmoid または ReLU)1本。

### 9.6 可視化(viz)

**core 側が汎用ビューを提供し、各環境はそれに描画データを渡すだけ**にする(環境ごとにフル可視化を実装させると負担が重いため)。初版が備えるべきビュー(行137-140):

- 環境の状況モニタ(適切な指標とその時系列)。
- 情報処理系の可視化(構造、活動 ── リアルタイムまたは部位ごとの活発度)。
- 系統進化の可視化(系統樹)。
- 人間および LLM が解釈可能な事象ログ。
- 3D/4D/抽象空間の可視化は初版では無理せず、**2D(走光性は実質1D)から始める**。

### 9.7 計算リソースの配分

リソースは情報処理系インスタンスごとに**等しく配分**したい(ネットワークを拡大したもの勝ちを防ぐ)。ただし利用可能リソースは使い切りたいので、一律に抑制して平等化する方法は採らない。

- **時間ベースの配分は却下**(「1秒で区切る」と、1秒で更新が終わらないインスタンスが全く動作できなくなる。時間は環境依存で再現性もない)。
- **演算回数ベース(ノード更新回数・エッジ評価回数のバジェット)で区切る。** 遅いマシンでも速いマシンでも公平で、決定的(再現可能)。
- バジェット N の具体値と超過時の扱いは §8 で保留。初版ダミーは「フレームだけ起動」(§7.2-4)。

### 9.8 並行機構(concurrency)

§2.2/§2.3 の制約 ── **両クロックがロックステップにならないこと、buffer interface を transport 非依存に保つこと** ── を満たす限り、プロセス/スレッドのいずれを使うかは Claude Code の裁量。

- 初版ネットワークは小規模(数百ノード)で、求めるのは並列高速化ではなく**クロックの論理的独立**であるため、最小で済む機構でよい。
- ただし将来の分散化(別マシンへの展開)を見据え、buffer 境界が並行機構の詳細を漏らさないこと。これさえ守れば、後でプロセス+ネットワーク buffer に移行できる。

### 9.9 CLI / UI

- **初版は CLI + 単一の Web UI に絞る**(ネイティブ GUI は作らない ── 三重メンテは年単位プロジェクトの負債になる)。
- **CLI で全機能を叩ける**こと。サブコマンド構成(例: `run` / `init` / `validate` / `resume`)。依存を最小化するなら標準ライブラリ `argparse` のサブパーサで足りる。CLI が複雑化する場合に限り `click` 等の採用を許容する(判断は Claude Code に委ねるが、過剰な依存は避ける)。
- **可視化は Web UI に集約**(自己ホストしやすく、ブラウザ1つで完結)。初版はまず**保存済みスナップショット/ログからの可視化**で足り、リアルタイム描画は発展課題としてよい。フロントエンドはビルド工程を持たない構成(素の HTML/JS + 軽量な描画ライブラリ)を優先し、保守を軽くする。バックエンドを置く場合も軽量なものに留める。

### 9.10 テスト

3層に分ける。`pytest` を基盤とする。

1. **決定論的な単体テスト**: 座標計算・距離計算・genome の変異/配合が仕様どおりか。
2. **不変条件テスト(property-based, `Hypothesis`)**: 接続点が必ず `[-1,+1]` に収まるか、リソース予算を超えないか、ノード数が上限内か。
3. **進化が機能しているかの統計的テスト**: 「世代を経て平均適応度が伸びるか」を緩い回帰テストとして見る。確率的なので厳密な pass/fail にはできず、「明らかに壊れていないか」の検出が目的。

### 9.11 ログ

- **システムログ**: 標準ライブラリ `logging`。十分な粒度で。
- **事象ログ**: 環境内で何が起きたかを、**人間と LLM の両方が読める形式**(JSONL 推奨)で記録(行137)。

### 9.12 ドキュメント / コード品質

- **docstring は必須**。
- 型ヒント・docstring・明示的な例外を最低ラインとする(LLM が後から保守する際の可読性のため)。
- コーディングスタイルの細部は Claude Code の裁量だが、上記の最低ラインは満たすこと。

## 10. 拡張可能な構造への要求(新版追加の土台)

**環境や発達ルールの「新版追加手順書」は、本仕様では作らない。** 手順書はインターフェース契約の取扱説明書であり、その細部(メソッド名・引数の型・戻り値の正確な形)は**初版を実装して契約が実物として固まった後**に、実物を見ながら別セッションで書くのが最も正確だから(想像で先に書くと実装と乖離し、最も腐りやすいドキュメントになる)。

ただし、**手順書を後で書けるための土台は初版で作る**こと:

1. **environments と development のインターフェースを明示的な抽象基底/Protocol として定義する**(§5 冒頭)。その定義自体が「新版が満たすべき契約」の一次資料になる。
2. **ダミー実装を参照実装として位置づけ、十分なコメントと docstring を付す**(§7)。新版を作る者はまずこれを読んで真似る。
3. **各 plugin ディレクトリに README とモジュール docstring の枠を最初から置く**(中身は薄くてよい。場所だけ確保し、後で実物の契約が固まったら埋める)。

## 11. 初版の完成条件(Definition of Done)

以下がすべて満たされたとき、初版(骨格)は完成とみなす。

- [ ] orchestrator が設定(TOML)を読み、ダミー環境(走光性)とダミー発達ルールで**試行を一周以上回せる**。
- [ ] 接続点が buffer 経由で双方向に流れ、値が常に `[-1,+1]` に収まる(クランプが効いている)。
- [ ] 生死判定が環境側で起き、orchestrator に通知される。
- [ ] 連続適応度が算出され、終了条件(全員満腹/全員死亡)で試行が打ち切られる。
- [ ] genome がシードを含み、片親コピーで継承され、変異が適用される。報酬系ノード(1つ)が genome 規定の位置に固定配置される。
- [ ] ノードとエッジの追加・削除の両経路が、ダミーでも一度は実行される。
- [ ] セッションの保存・再開ができる。スナップショットが JSONL で人間可読に保存される。
- [ ] システムログと事象ログが出る。
- [ ] CLI から全機能(run / init / validate / resume 等)を叩ける。
- [ ] Web UI で、保存データから 環境モニタ・ネットワーク構造・系統樹 が見られる。
- [ ] テスト3層(単体・不変条件・統計)が用意され、通る。
- [ ] core / environments / development / viz の二層分離が守られ、各 plugin ディレクトリに README 枠と抽象基底がある。

**初版完成後の次フェーズ**: ①発達ルールの数学設計(§8 の空き枠を埋める)、②新版追加手順書の作成(§10 の土台から、実物の契約を見て起こす)。これらは独立した後続セッションで行う。

## 付記: プロジェクト名

`Graphian` は仮称。情報処理系をグラフとして捉える本質には合うが、グラフ理論・グラフDB 等と検索衝突しやすく、本プロジェクト固有の核心(同心円構造・発達ルールの進化・生存駆動)を指してはいない。名前は最後にいつでも変えられるため、実装上の判断には影響しない。現状は `graphian` をパッケージ名・ディレクトリ名に用いる。
